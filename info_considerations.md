# Project Considerations

A candid review of the Competitive Intelligence Agent project: What works well, what trade-offs are baked in, what limitations to be aware of, and where there is clear room for improvement. Covers the parent agent (`agent.py`, `agent_modes.py`, schedule runner, dashboard, tests) and the [`evaluation/`](evaluation/) subproject.

All scripts are explicitly **Learning/Demo status**, so the items below are framed as design notes rather than production blockers.

---

## Strengths

### Architecture and structure
- **Symmetric layout between parent and harness.** The `evaluation/` subproject mirrors the parent's structure file-for-file (runner, config, models, persistence, dashboard, tests), see table in [`info_project.md`](info_project.md).
- **Single source of truth for evaluation metrics.** Every metric is declared once as a `MetricDef` in [`evaluation/eval_metric_registry.py`](evaluation/eval_metric_registry.py), the SQLite schema, CSV headers, dashboard cards, and run-summary averages all derive from it. Adding a metric is a one-line registry change plus a compute function.
- **Centralised configuration.** All runtime tuning lives in [`config.py`](config.py) (agent) and [`evaluation/eval_config.py`](evaluation/eval_config.py) (harness), including domain filters, search depth, schedule, judge model, thresholds, concurrency. No tuning is buried in business logic.
- **Clean separation of execution modes.** [`agent_modes.run_manual_chat`](agent_modes.py) (REPL) and [`agent_modes.run_automated_lookout`](agent_modes.py) (batch) sit on top of one [`agent.run_agent`](agent.py) coroutine, keeping the supervisor pipeline single-sourced.

### Agent design
- **Delta-based significance scoring.** [`system_prompt_agent.txt`](system_prompt_agent.txt) scores news by *newness vs. PREVIOUS STATUS*, not absolute importance. A deliberate choice that prevents repeated stories from re-flagging.
- **History injection via `config['configurable']`, not the user message.** [`_build_supervisor_prompt`](agent.py) pulls `history_context` from runnable config so sub-agents never receive history (token savings + cleaner sub-agent context). Documented in the function docstring.
- **Runtime-assembled, callable prompts.** [`_build_supervisor_prompt`](agent.py) is a function, not a static string. `create_supervisor` calls it on every invocation, pulling per-case variation from `config['configurable']` and concatenating it onto a compact base prompt loaded from [`system_prompt_agent.txt`](system_prompt_agent.txt). Sub-agents follow the same pattern. Benefits:
  - **Token efficiency.** Base prompt is shared across calls. Only per-case variation differs, which means it does not have to pre-encode every possible scenario in advance.
  - **Cold-start handling.** Empty/whitespace `history_context` is normalized to `"PREVIOUS STATUS: No prior data."` so the agent always has an explicit marker to compare against — relevant for L2 live runs which run cold-start by design.
  - **Caller decoupling.** Manual chat, scheduled lookout, and the eval harness each populate `config['configurable']` differently; the supervisor doesn't know or care which mode is invoking it.
  - **Eval-harness compatibility.** L2 live deliberately strips `history_context` to test cold-start; the eval just passes a different config, no prompt-string surgery needed.
  - **Forward extensibility.** Adding a new per-call field (date, focus directive, sentiment override) is one config key + one line in the callable, no prompt rewrite or graph recompile.
- **STOP RULE + DIRECTIVE prefix.** The supervisor prompt strictly forbids more than one tool call per query and routes via an explicit DIRECTIVE prefix in [`agent_modes.py`](agent_modes.py). Both are enforced in evaluation by `directive_compliance` and `tool_call_count` gates.
- **`output_mode="last_message"` on the supervisor.** Raw search results don't bloat the supervisor's context, only the sub-agent's final synthesis is passed back (see [`agent.py:supervisor`](agent.py)).
- **Structured sentiment extraction.** Sentiment uses [`SentimentResult`](pydantic_models.py) via `with_structured_output` instead of regex, while the SIGNIFICANCE_SCORE itself is regex-extracted (see Limitations to note).
- **Per-company history is injected from a durable SQLite ledger.** [`get_latest_company_intel`](memory_ledger_db.py) reads the last N reports for a company from `intel_ledger` and [`run_automated_lookout`](agent_modes.py) injects them as a `history_context` string into the next prompt. Persistence is structured (one row per investigation, queryable from the dashboard) and explicit. Knowledge enters the agent through one named channel, not implicit graph state.
- **No `MemorySaver` checkpointer on the supervisor — by design.** Scouts are one-shot per company, not conversational. [`agent_modes.run_automated_lookout`](agent_modes.py) sets a per-company `thread_id` on the runnable config, but [`agent.py`](agent.py) compiles the supervisor without a checkpointer (see comment at [`agent_modes.py:124-126`](agent_modes.py)), so each run starts from a clean graph state. The `thread_id` plumbing is left in place so a future conversational mode could drop a `MemorySaver` in without touching the caller.

### Evaluation harness
- **Three execution variants for cost-controlled testing.** [`eval_config.RUN_VARIANTS`](evaluation/eval_config.py): `l1` (sub-agent isolation, live), `l2_fixture` (full panel against hand-authored fixtures, no search APIs), and `l2_live` (real end-to-end). L2 fixture mode lets judge-prompt iteration happen without burning search budget.
- **Token tracking on judges.** [`_TokenTrackingJudgeLLM`](evaluation/eval_engine.py) accumulates input/output tokens across the entire DeepEval panel, so per-case judge cost is visible in the dashboard.
- **Verdict reasons are collected, not short-circuited.** [`EvaluationEngine._compute_status`](evaluation/eval_engine.py) records *every* failing gate per case so the dashboard can show the full failure list without re-running.
- **SQLite ledger is registry-derived and add-only.** [`eval_sqlite.py`](evaluation/eval_sqlite.py) auto-migrates new metric columns on startup. Gate thresholds are persisted with each run so the dashboard can warn on threshold drift.
- **Session grouping.** All variants run in one `eval_runner.py` invocation share a `session_id`, making cross-variant comparison in [`eval_dashboard.py`](evaluation/eval_dashboard.py) trivial.
- **Robust supervisor-name diagnostic.** [`_run_agent_live`](evaluation/eval_engine.py) records observed `on_chain_end` event names and emits an actionable error if `_SUPERVISOR_EVENT_NAME` is never seen. A concrete guard against silent breakage when `agent.py` is renamed.

### Test coverage
- **111 unit tests** across [`tests/`](tests/) (parent: schemas, persistence, helpers) and [`evaluation/tests/`](evaluation/tests/) (45 + 53 + 13 — utils, deterministic metrics, registry invariants).
- The registry invariant tests in [`tests/test_eval_metric_registry.py`](evaluation/tests/test_eval_metric_registry.py) catch most "drifted in one place but not the other" bugs at test time.

### Operational hygiene
- **Single shared logger.** [`utils.setup_logger`](utils.py) returns the same configured logger across modules. Idempotent on repeated calls.
- **Auto-refreshing dashboards.** Both [`dashboard.py`](dashboard.py) and [`evaluation/eval_dashboard.py`](evaluation/eval_dashboard.py) watch the underlying DB's max-timestamp and bust their caches only when new rows arrive.

## Trade-offs

- **Verdict gate thresholds are global.** [`eval_config.JUDGE_THRESHOLD`](evaluation/eval_config.py), `REQUIRED_KEYWORD_THRESHOLD`, `MIN_REPORT_LENGTH` apply to every metric / every case. Per-case overrides are not currently supported.
- **Flat-folder layout instead of installable packages.** Neither the parent project nor [`evaluation/`](evaluation/) is set up as a `pip`-installable package. There is no `pyproject.toml` or `__init__.py`. Test imports are made to work via [`tests/conftest.py`](tests/conftest.py) and [`evaluation/tests/conftest.py`](evaluation/tests/conftest.py), each of which prepends its parent folder to `sys.path` at pytest collection time. Pro: No build system, no install step, every script is directly runnable. Con: The trick only fires under pytest. Running a test file directly with `python tests/test_*.py` raises `ImportError`, and the modules cannot be imported from outside their own folder without callers replicating the same `sys.path` adjustment.
- **Only fully-enabled L2 runs are persisted to the ledger.** [`eval_sqlite.save_run`](evaluation/eval_sqlite.py) early-returns when the run's `enabled_groups` isn't the full set (`judge`, `format`, `routing`). Partial-group runs (e.g. `judge`-only iterations on prompt wording) still produce JSON+CSV artifacts in `evaluation_results/`, but no row goes into `eval_ledger.db`. L1 runs bypass this via [`save_l1_run`](evaluation/eval_sqlite.py) because L1 metrics are always-on by definition (no judge/format/routing toggle exists for L1). Pro: The Historical Trends tab always compares apples-to-apples, the pass-rate line isn't polluted by partial runs whose gates were selectively switched off. Con: Partial runs don't show up in the dashboard at all. Iterating on judge prompts means reading the JSON/CSV directly, or running all three groups even when only one is being changed.
- **Composite metric pattern for `keyword_checks`.** [`eval_metric_registry.py`](evaluation/eval_metric_registry.py) treats `keyword_checks` as a composite metric: the per-case computation returns a dict (`{"required_keyword_hit_rate": ..., "disallowed_keyword_hits": ...}`), and the registry carries a `composite=True` flag plus three paired-tuple fields (`composite_sql_columns`, `composite_summary_avg_keys`, `composite_labels`) that consumers expand into flat SQL columns, run-level averages, and dashboard labels. Two flat scalar `MetricDef`s could have produced the same DB schema and the same dashboard output with strictly less plumbing.

  - **Grouping mirrors the computation.** [`compute_keyword_checks`](evaluation/eval_metrics.py) does one regex pass and emits two related numbers, composite keeps them grouped under one key in the per-case result so they travel together in JSON exports and stay visually adjacent in the dashboard. Toggle/group state stays in sync by construction (one MetricDef, one toggle) instead of by convention across two separate MetricDefs.

  - **Future expansion is cheap.** Adding a third keyword sub-metric (e.g. `forbidden_phrase_count`) is one tuple entry in each of the three composite fields, no new top-level MetricDef, no new toggle wiring. The same pattern absorbs additional composite domains (`citation_checks`, `format_checks`) without further machinery.

  - **Cost: ~40–50 lines of generic plumbing for one specific case.** A dedicated [`composite_summary_avg_specs`](evaluation/eval_metric_registry.py) helper, two `if metric.composite` branches in [`case_sql_columns`](evaluation/eval_metric_registry.py) and [`dataframe_columns_for_group`](evaluation/eval_metric_registry.py), and a separate composite loop in [`eval_report_manager.py`](evaluation/eval_report_manager.py) (with nested `result_dict.get(composite_key, {}).get(sub_key)` access) all exist to support one composite metric. Two flat scalars would have used the existing scalar-only path everywhere.

  - **Break-even.** The composite pattern starts paying for itself at ≥3 sub-metrics under one domain, or ≥2 composite domains. With only `keyword_checks` at 2 sub-fields, the machinery is borderline over-engineered today, but the framework is in place, so any next composite addition is nearly free.

- **Supervisor + sub-agents instead of a single ReAct agent or a scheduled scout pipeline.** [`agent.py`](agent.py) uses [`create_supervisor`](agent.py) to orchestrate two sub-agents (`brave_scout`, `tavily_analyst`), each with its own dedicated tool, prompt, and budget. Two alternatives were rejected: (a) A single ReAct agent with both Brave and Tavily as tools. (b) A scheduled scout that runs independently and feeds a downstream agent with its own deep-search tool, which then performs the deep dive and writes the report.

  - **Token consumption.** Both alternatives have lower fixed orchestration overhead per query. Alt A skips the supervisor's routing decision entirely, Alt B's scout runs once per schedule tick rather than once per query. The supervisor here costs one extra LLM call per invocation. Two effects offset this: `output_mode="last_message"` keeps raw search output out of the supervisor's context (only the sub-agent's compact synthesis is passed back), and on a watchlist of N companies × M invocations per day the orchestration overhead is small compared to the search-API spend.

  - **Extension to additional sub-agents.** Adding a third sub-agent (SEC analyst, press-release analyst, etc.) is one new sub-agent definition + one DIRECTIVE prefix, the supervisor's routing logic doesn't change shape. Alt A would require adding tools to the same ReAct agent and rewriting its prompt to accommodate the new role, which compounds. Alt B would require a new scheduled job + new downstream agent, heavier than adding a sub-agent.

  - **Per-call scout customization.** Each company on the watchlist has its own `special_focus`, and the supervisor prompt receives this plus a DIRECTIVE via `config['configurable']`, so the scout can be told: "For Apple, watch AI hardware", or "for OpenAI, watch partnerships" on a per-call basis. Alt A reads the same config and works identically. Alt B's scheduled scout cannot see per-call config (there is no per-call config layer for a scheduled job), so per-company customization of *what the scout looks for* would require a separate orchestrator feeding focus directives into the scout, which starts to look like a supervisor pattern again.

  - **Routing observability.** "Routing" here means deciding which sub-agent runs for a given query, either `brave_scout` for cheap triage, or `tavily_analyst` for expensive deep dives. With supervisor + sub-agents, that decision is **explicit**: The supervisor emits a named handoff tool call (`transfer_to_brave_scout` or `transfer_to_tavily_analyst`) that is directly visible in the event stream, so the eval harness's [`directive_compliance`](evaluation/eval_metrics.py) gate can read the event name and instantly grade whether the agent obeyed the `DIRECTIVE:` prefix in the query. With Alt A, routing is **implicit** in tool-call decisions: A single ReAct agent does not emit a named "route" step, it just picks Brave or Tavily as a regular tool call, so the harness would have to infer intent from raw tool traces. With Alt B, there is **no per-query routing decision at all**: The architecture hard-codes which agent runs when, so `directive_compliance` has nothing to grade.

  - **Per-agent prompt isolation.** Alt A would encode tool selection, per-tool query construction, and report formatting in one prompt, that is long and brittle. Alt B has separate scout and analyst prompts by design, so this aspect mostly differentiates from Alt A. Here, [`_brave_prompt`](agent.py) and [`_tavily_prompt`](agent.py) are independently tuned for cheap triage vs. expensive deep-dive.

  - **Per-agent tool budgets.** The STOP RULE ("at most one tool call per query") applies cleanly per sub-agent invocation. Alt A would have to express this as a shared budget across two tools in one ReAct loop, which is awkward because the decision tree is shared. Alt B has separate budgets by design, similar to the supervisor + sub-agents design here.

- **Sub-agents are prebuilt with `create_agent` and handed to `create_supervisor`, not built as explicitly-compiled subgraphs.** In [`agent.py`](agent.py), `brave_agent` and `tavily_agent` are produced by [`langchain.agents.create_agent`](agent.py) and passed straight into [`create_supervisor`](agent.py), only the supervisor itself is explicitly compiled. An alternative would be to build each sub-agent as a custom LangGraph `StateGraph` with its own state schema, compile each one explicitly, and compose them into the supervisor as named subgraphs (either via `create_supervisor` accepting compiled graphs, or by hand-rolling the supervisor with `StateGraph.add_node` and explicit edges).

  - **Pros 1.** Each sub-agent becomes an explicit compiled unit with a state schema you define, rather than inheriting `MessagesState` and the ReAct loop shape that `create_agent` bakes in. Inputs and outputs at the sub-agent boundary are typed and named, so the contract between supervisor and sub-agent is explicit instead of conventional.

  - **Pros 2.** Modularized testing and troubleshooting also benefit. With a typed state schema you control, tests assert on named output fields instead of parsing `messages[-1].content`, and the supervisor → sub-agent boundary is inspectable in clearly typed terms instead of as opaque message lists.

  - **Cons.** Each sub-agent grows from a ~10-line `create_agent` call to ~30–50 lines of `StateGraph` wiring. You lose `create_agent`'s built-in tool-calling protocol and message-state plumbing.

## Limitations to note

- **Windows-specific stdout reconfiguration at import time.** [`agent.py:15-16`](agent.py) calls `sys.stdout.reconfigure(encoding="utf-8")` unconditionally. Harmless on Linux/macOS, but tied to a CP1252-on-Windows quirk.
- **SIGNIFICANCE_SCORE extraction is regex-based.** [`agent.run_agent`](agent.py) parses the score with `re.search(r"SIGNIFICANCE_SCORE\D+(\d+)", text)`. Reliable in practice because the system prompt mandates a fixed trailer, but a malformed report silently records `significance_score=0`. Sentiment extraction (later in the same function) already uses structured output via `with_structured_output(SentimentResult)`, the score could be migrated to the same pattern.
- **Brave domain filters are prompt-based, not API-native.** Tavily applies `EXCLUDED_DOMAINS` / `INCLUDED_DOMAINS` natively, but [`agent._brave_prompt`](agent.py) instructs Brave via natural language (`"Never include results from: …"`). Compliance is best-effort. Layer 1's `domain_exclusion_respected` metric in [`evaluation/eval_metrics.py`](evaluation/eval_metrics.py) measures the actual hit rate.
- **L2 fixture mode is synthetic.** Hand-authored `fixture` blocks in [`eval_cases_layer2.json`](evaluation/eval_cases_layer2.json) are paired with synthetic `history_context` and `expected_answer_points`. They are great for iterating on judge prompts cheaply, but they can drift from real production outputs over time. Keeping a periodic `l2_live` run guards against fixture drift.
- **`_SUPERVISOR_EVENT_NAME = "agent"` is a runtime-checked string.** [`evaluation/eval_engine.py:115`](evaluation/eval_engine.py) hardcodes the LangGraph chain-end event name and emits a diagnostic if it's never seen. Renaming the compiled graph requires updating this constant. The diagnostic surfaces the issue cleanly, but the coupling is implicit rather than enforced at import time.
- **Schedule "interval mode" defaults to 2 minutes.** [`config.SCHEDULE_INTERVAL_MINUTES = 2`](config.py) is fine for demo, but every run hits Brave/LLM-model for every company on the watchlist. Easy to forget when leaving `launch_schedule_runner.py` running.
- **`dashboard.py` only displays the columns it queries.** Adding a new column to [`memory_sqlite3.intel_ledger`](memory_sqlite3.py) requires updating both the `CREATE TABLE` and the `SELECT` in [`dashboard.py:54-57`](dashboard.py). The eval ledger has a registry, but the parent ledger does not.
- **No retry/backoff in `run_agent`.** A transient Brave/Tavily failure raises through to the caller. The lookout loop catches and continues per-company, but the failed company gets nothing in the ledger for that run.

## Possible future improvements

- **Switch SIGNIFICANCE_SCORE extraction to structured output.** The same `with_structured_output` pattern already used for `SentimentResult` in [`agent.py`](agent.py) would eliminate the regex branch and give a clear "missing score" failure mode.
- **Add a parent-side metric/column registry.** Today, [`memory_sqlite3.py`](memory_sqlite3.py), [`memory_ledger_db.py`](memory_ledger_db.py), and [`dashboard.py`](dashboard.py) each independently know the column list. Mirroring the eval registry pattern would make ledger schema changes a one-place edit.
- **Native Brave domain filters where possible.** If/when [`langchain_community.tools.BraveSearch`](agent.py) exposes domain-filter parameters, swap the prompt-based instruction in [`_brave_prompt`](agent.py) for the native version to make filtering a hard guarantee instead of best-effort.
- **Add an integration test for `run_agent`.** The parent [`tests/`](tests/) suite covers the schemas and the persistence layer in isolation. An end-to-end smoke test that mocks Brave/Tavily and runs `run_agent` would catch breakage in the streaming/event-handling code path and in the supervisor wiring (callable-prompt contract, handoff parameter interactions) that the unit tests miss.
- **Log rotation.** [`utils.setup_logger`](utils.py) attaches a plain `FileHandler` to `agent.log`. For a long-running scheduled process, swapping in a `RotatingFileHandler` would prevent unbounded log growth.
- **Surface judge-call retries.** [`EvaluationEngine._run_judge_metric`](evaluation/eval_engine.py) treats the first exception as a hard failure on the metric. A short retry on transient errors (rate limits, network) would reduce REVIEW noise that isn't actually about the agent.
- **Tighten the registry abstraction in [`eval_report_manager.py`](evaluation/eval_report_manager.py).** One usage at line 65 (`{m.key for m in METRICS if m.summary_avg_key}`) reaches into the raw `METRICS` list — the lone consumer outside the registry that does so. Promoting this to a `keys_with_summary_avg()` helper in [`eval_metric_registry.py`](evaluation/eval_metric_registry.py) would close the abstraction leak and let the consumer drop the `METRICS` import entirely, leaving the registry's helper functions as the only public surface.
- **Rename the local `metrics` dict in [`eval_engine.py`](evaluation/eval_engine.py).** The local variable holds per-case computed values, but shares its name with the registry's "metrics" terminology (`METRICS` list, `MetricDef`, etc.). Something like `case_metric_values` or `computed_metrics` would disambiguate the value-dict from the registry-level concept at a glance. Local-variable rename, low risk.
- **Rename `tool_call_count` → `supervisor_tool_call_count`.** The metric is computed from `supervisor_tool_names` only ([`eval_engine.py:838-840`](evaluation/eval_engine.py#L838-L840)), so it counts supervisor-level handoff calls, not all tool calls in the run. The longer name would make the scope explicit and prevent future readers from interpreting it as a run-wide tool-call total. Note: this is a registry-key change, so it triggers the rename / save-aside workflow documented in [`evaluation/README.md`](evaluation/README.md) — the cost is a single deliberate save-aside, then all consumers (DB schema, CSV, dashboard cards) auto-update on next startup.
- **Document the rename / save-aside workflow more visibly.** [`evaluation/README.md`](evaluation/README.md) covers it under "Schema Policy", and [`eval_sqlite.py`](evaluation/eval_sqlite.py) covers it in the module docstring, but a developer renaming a metric for the first time is more likely to read [`eval_metric_registry.py`](evaluation/eval_metric_registry.py). A one-line comment there pointing to the policy would close the gap.
- **Multi-ledger DB browser/selector for the dashboard.** [`eval_dashboard.py`](evaluation/eval_dashboard.py) reads from a single hardcoded `eval_ledger.db`. When a metric rename triggers a save-aside (per Schema Policy), the archived DB becomes invisible to the dashboard. A small module that scans the `evaluation/` folder for `eval_ledger*.db` files, exposes them in a sidebar dropdown, and lets the user select which one to view would unlock the value of the save-aside workflow. Historical schemas remain browsable instead of becoming cold storage.
- **Make `SCHEDULE_INTERVAL_MINUTES = 2` opt-in, not the default.** Either default the file to cron mode (`SCHEDULE_INTERVAL_MINUTES = 0`) or guard the interval mode behind an explicit `--interval` CLI flag in [`launch_schedule_runner.py`](launch_schedule_runner.py) so an accidentally-left-running process doesn't burn budget.
- **Decompose a sub-agent into multiple internal nodes if it grows multi-step logic.** Today each sub-agent is a single-step ReAct loop: One LLM call, one tool call, one report. If a future sub-agent needs genuine multi-step internal logic (e.g., `search → re-rank → filter → synthesize` as distinct LLM-call nodes with intermediate state), that one sub-agent can be promoted to a custom LangGraph `StateGraph` and added to the supervisor as a subgraph while the others stay as `create_agent`. Per-internal-node observability and unit tests (each node fires its own `on_chain_start` / `on_chain_end` in `astream_events`, each is independently invocable) come along once the decomposition exists.
